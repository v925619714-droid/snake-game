Разрешено правилами Apple/Google (стандартный паттерн, экран «обновите, чтобы продолжить»).

## 2. Что мы имеем (проверено по коду)

**Козырь: вся игровая симуляция — чистый TypeScript.** `src/game/{logic,duel,party,rating,bot,partyBot,economy}.ts` не импортируют ни `react-native`, ни `expo`, ни `AsyncStorage` (только внутримодульные импорты из `./logic`, `./skins`). Значит `duelStep/partyStep/duelTurn/partyTurn/duelNewMatch` можно **дословно запустить на Node-сервере** и считать матч авторитетно. Это ядро всего переезда realtime.

**Поверхность Supabase в клиенте (что переписывать) — узкая и понятная:**
- **Auth** (`src/lib/auth.ts`): `signInAnonymously`, `signInWithOtp`, `verifyOtp`, `getSession`, `getUser`, `signOut`, `rpc('delete_my_account')`.
- **Data** (`src/lib/leaderboard.ts`): `from('profiles').select(...)` (топ + профиль по id) + 4 RPC: `submit_match`, `upsert_profile`, `upsert_wallet`, `delete_my_account`.
- **Realtime** (`src/net/useRoom.ts`, `src/net/usePartyRoom.ts`): 4 канала (`room-<code>`, `matchmaking`, `mm-ranked`, `party-<code>`), события broadcast `state/input/ping/resync/pstart/pstate/pinput/pstake/phello` + presence.
- **Клиент**: `@supabase/supabase-js`, сессия в AsyncStorage, JWT autorefresh. Env: `EXPO_PUBLIC_SUPABASE_URL/KEY`, `EXPO_PUBLIC_POSTHOG_KEY/HOST`.

**Схема БД и логика — в `supabase/profiles.sql`** (одним файлом): таблицы `profiles` (+ кошелёк-колонки + анти-чит: `last_match_at`, `day_gain`, `day_gain_date`), `matches` (журнал). RLS: чтение всем, запись **только через SECURITY DEFINER функции** (нельзя править чужую строку). `profiles.id = auth.uid()`.

**Важно: серверный анти-чит УЖЕ есть** (T61 в `submit_match`): берёт реальный рейтинг соперника из БД по его uid (клиентский `oppRating` — только для бота), K=32, кламп ±32, кулдаун 6с, дневной кап прироста 250, отклонение self-матча, журнал в `matches`. Остаточный риск (клиент сообщает исход) помечен в SQL — закрывается двойным подтверждением позже. **Эту логику переносим как есть** (замена `auth.uid()` на uid из JWT).

## 3. Целевая архитектура

Один **отдельный headless-сервер** (новый HostKey CPU-VPS, НЕ дизайнерский 37427), всё в Docker Compose:

```
                    Интернет
                       │
              ┌────────┴────────┐
              │  Caddy (TLS,     │  ← Let's Encrypt авто
              │  reverse proxy)  │
              └───┬─────┬────┬───┘
      api.*/auth.*│     │    │ rt.*  (wss)
        ┌─────────┘     │    └──────────┐
        ▼               ▼               ▼
  ┌───────────┐  ┌───────────┐  ┌──────────────────┐
  │ PostgREST │  │  GoTrue   │  │ Игровой сервер    │
  │ (.from/   │  │ (auth:    │  │ Colyseus (Node+TS)│
  │  .rpc)    │  │ аноним+   │  │ реиспользует      │
  └─────┬─────┘  │  OTP)     │  │ src/game/*        │
        │        └─────┬─────┘  │ авторитетный loop │
        └──────┬───────┘        │ + дельты + анти-  │
               ▼                │ чит + reconnection│
        ┌─────────────┐         └────────┬─────────┘
        │ PostgreSQL  │◄─────────────────┘ (submit_match и т.п.)
        │ 16          │
        └─────────────┘
               │ ночной pg_dump
               ▼  (оффсайт на ПК, как бэкап графа)
```

**Домены** (у нас есть skillmake.ru через Timeweb, добавляем сабдомены → IP нашего сервера):
- `api.snake.skillmake.ru` → PostgREST (data)
- `auth.snake.skillmake.ru` → GoTrue
- `rt.snake.skillmake.ru` → игровой сервер (wss)
- (веб остаётся на `snake.skillmake.ru` → GitHub Pages, не трогаем)

## 4. Ключевые технические решения (с обоснованием)

**Р1. Data + Auth: self-host ядра Supabase (Postgres + GoTrue + PostgREST), а НЕ писать свой API с нуля.**
Почему: клиент уже написан под `supabase-js`. Если поднять self-hosted PostgREST + GoTrue (официальный docker-compose Supabase), то `auth.ts` и `leaderboard.ts` **работают почти без изменений** — меняется только URL/anon-ключ в env. Схема (`profiles.sql`) и RPC переносятся дословно. Плата за сообщения/соединения у Postgres/GoTrue/PostgREST **отсутствует** (в отличие от Realtime) — то есть проблема стоимости для data/auth снимается самим фактом self-host.
Альтернатива (свой lean REST + custom JWT/OTP) — меньше сервисов, но переписывать `auth.ts` + `leaderboard.ts` целиком + мигрировать uid. Отложить как «фаза 2 оптимизации».

**Р2. Realtime: заменить на свой авторитетный игровой сервер (Colyseus, Node + TS).**
Почему именно это переписываем (а не self-host Supabase Realtime): (1) Realtime — единственный источник дорогой платы за соединения/сообщения; (2) текущий нетокод host-authoritative без интерполяции и с остаточным чит-риском. Свой сервер на **Colyseus** даёт из коробки: авторитетные комнаты (сервер крутит `duelStep/partyStep`), **дельта-синхронизацию состояния** (= решает проблему объёма сообщений by design), matchmaking, reconnection, presence. Дуэль = комната на 2, Office Royale = комната на 5–10. Реиспользуем `src/game/*` как модуль симуляции.
Один заход закрывает **три пункта бэклога сразу**: уход от дорогого Realtime + серверная авторитетность (анти-чит) + интерполяция против лагов.
Альтернатива: тонкий `ws`-сервер, зеркалящий текущий broadcast-протокол — меньше фреймворка, но дельты/reconnection/rooms писать руками. Colyseus предпочтительнее.

**Р3. Оркестрация — Docker Compose** (postgres + gotrue + postgrest + gameserver + caddy). Воспроизводимо, один `docker compose up`, легко переносить между боксами.

**Р4. TLS — Caddy** (авто Let's Encrypt, минимум конфигурации) вместо nginx+certbot.

**Р5. Email для OTP — существующий SMTP** (Unisender Go / Timeweb, реквизиты в SECRETS) в конфиге GoTrue. Свой почтовик не поднимаем.

## 6. Сервер: состав и настройка

- **ОС**: Ubuntu 22.04/24.04 LTS **headless** (не CentOS-десктоп как 37427). Отдельный аккаунт/бокс.
- **Класс**: для старта достаточно **4 vCPU / 8–16 ГБ / SSD**; с запасом под рост — 8 vCPU / 16–32 ГБ. GPU **не нужен** (для релея+БД простаивает — берём CPU-VPS, не vGPU).
- **Docker Compose сервисы**: `postgres:16`, `gotrue`, `postgrest`, `gameserver` (наш образ Node+TS+Colyseus), `caddy`.
- **Сеть/файрвол (UFW)**: наружу только 443 (Caddy) + 22 (SSH, ограничить по IP/ключу). Внутренние сервисы — только в docker-сети, наружу не торчат.
- **DDoS/устойчивость**: Cloudflare-проксирование сабдоменов (оранжевое облако) для api/auth (HTTP) — бесплатный слой защиты; для `wss` realtime — Cloudflare поддерживает WebSocket, либо оставить напрямую с rate-limit в Caddy. `fail2ban` на SSH.
- **⚠️ Грабля HostKey**: банят за长 100% CPU (майнинг-паттерн). Игровой loop держать эффективным (тик, дельты), не пережигать ядра; мониторить.

## 7. Миграция данных

1. **Схема**: адаптировать `profiles.sql` — заменить `auth.uid()` на функцию получения uid из JWT (GoTrue self-hosted даёт тот же `auth.uid()` — тогда правок почти нет). Применить к своему Postgres.
2. **Данные**: `pg_dump` из Supabase cloud (`profiles`, `matches`, `auth.users`) → restore в свой Postgres. **Сохранить uid** (иначе `profiles.id` рассыплется) — GoTrue self-hosted позволяет импортировать `auth.users` с теми же id.
3. **Проверка**: контрольные суммы числа строк, топ-лидерборда до/после совпадает.

## 8. Стратегия перехода (dual-run — без «жёсткого рубильника»)

Приложение в App Store **зашивает URL бэкенда в бандл на этапе сборки** → старые установки ходят в Supabase cloud, пока пользователь не обновится.

- **Веб** (`snake.skillmake.ru`) — переключается **сразу**: пересобрать статик-бандл с новым URL → `deploy-web.ps1` → пользователи получают на следующей загрузке.
- **iOS** — только через **обновление приложения** (новый URL → EAS build → ревью Apple → пользователи обновляются). Старые версии продолжают жить на Supabase cloud.
- **Окно параллельной работы**: держим Supabase cloud живым, пока доля старых версий приложения не станет мала (**PostHog показывает распределение `app_version`**). Затем — финальная сверка данных (cloud → self-hosted, разово) и выключение Supabase.
- **Данные в окне**: новые результаты старых iOS-клиентов пишутся в cloud. Простейший путь — принять временное расхождение и слить один раз при закрытии cloud; либо периодический one-way sync cloud→self-hosted. Это самый тонкий операционный момент — время выключения выбираем по данным PostHog.

## 12. Риски и что решить Владимиру

**Риски:**
- **Единая точка отказа** (один бокс). Смягчение: бэкапы оффсайт + быстрый re-provision из docker-compose; при росте — второй бокс/реплика Postgres.
- **HostKey suspend за CPU** — держать loop эффективным, мониторить.
- **Окно dual-run и расхождение данных** — самый тонкий момент (раздел 8).
- **Ops-нагрузка** переходит к нам (бэкапы/TLS/мониторинг/патчи) — раньше это делал Supabase.

**Решить перед стартом:**
1. **Отдельный бокс под игру** (рекомендую) — заказываем новый headless CPU-VPS? (37427 остаётся дизайнеру).
2. **Realtime-фреймворк**: Colyseus (рекомендую) или тонкий `ws`?
3. **Auth**: self-hosted GoTrue (рекомендую, минимум правок) или свой JWT/OTP?
4. **Когда** стартуем: по плану развития переезд логичен, когда мультиплеер становится каналом роста (после апрува+вирусного спринта). Делаем сейчас или ставим в бэклог с готовым планом?
