# План развития Shake Work Off — синтез (Kimi + стратегическая оценка)

> Основано на: подробном анализе Kimi (`KIMI-ANALYSIS.md`, читала реальный код) + моя критическая оценка и приоритизация под текущую бизнес-ситуацию.
> Статус: **план на согласование, НЕ реализован.**

## 1. Оценка анализа Kimi

**Сильные стороны (согласен):**
- Диагностика техдолга точна: `App.tsx` = 1301 строка God-component — реальный тормоз.
- Верно выделила 4 корневые проблемы: техдолг, **нулевая монетизация**, слабый retention, мультиплеер на Broadcast без interpolation.
- Хорошая карта файлов — где что менять. Экономит время реализации.
- Уникальность игры (офисная ниша / командная виральность) отмечена как главный актив — это ключ.

**Где я расхожусь с Kimi по приоритетам:**
1. **Kimi недооценила текущий контекст: приложение ещё НЕ одобрено Apple** (REJECTED → переотправлено, WAITING_FOR_REVIEW). Пока нет апрува — вкладываться в крупные фичи рано (риск ещё правок под Apple + нет пользователей для валидации).
2. **Kimi предлагает много фич без данных.** Battle Pass, power-ups, режимы сложности, сезоны — это гипотезы. У нас есть PostHog: сначала данные (где отваливаются, какой режим живёт), потом строить. Иначе строим вслепую.
3. **Рефакторинг App.tsx Kimi ставит рано (Этап 1).** Но рефакторинг = 0 ценности для пользователя. Я бы делал его **инкрементально, под конкретные фичи** (когда касаемся файла), а не отдельным дорогим заходом на 2-3 дня.
4. **Rewarded ads для офисной аудитории — спорно.** Офисные играют украдкой в перерыв; реклама раздражает и «палит». Kimi ставит как quick win — я бы тестировал осторожно, а ставку делал на **B2B/командную монетизацию** (её же уникальный актив), которую Kimi отложила в Этап 2.
5. **Виральность у Kimi размазана** (шеринг в Этапе 0, рефералка в Этапе 1). Для офисной игры вирусный цикл — это ЯДРО роста, его надо собрать в один фокусный спринт, а не дробить.

## 2. Стратегическая рамка (моя)

**Позиционирование — не «ещё одна змейка», а «Snake для офисов и команд».** Всё развитие крутится вокруг цикла: *сотрудник зовёт коллег → команда играет в перерыв → победитель хвастается → приходят новые команды.* Это B2B2C-виральность, которой нет у конкурентов.

**Три принципа приоритизации:**
- **Сначала метрики роста и удержания, потом контент.** До монетизации нужен поток и retention.
- **Данные перед фичами.** PostHog-воронки → гипотезы → фичи под подтверждённую боль.
- **Техдолг — под фичи, не отдельно.** Рефакторим по мере касания.

## 3. Рекомендованный роадмап (по фазам, привязан к реальности)

### Фаза 0 — СЕЙЧАС (пока ревью Apple, код рискованно трогать)
Не рискуем крупными правками (могут понадобиться под Apple). Готовим почву:
- **Аналитика-воронки в PostHog:** onboarding→первая игра→вторая сессия→шеринг. Определить точки отвала. (низкий effort, высокий impact — это компас для всего плана)
- **Дизайн вирусного цикла** (реф-петля + shareable-карточка + Slack-сообщение) — спроектировать, не кодить.
- **Android-подготовка** (eas.json android, иконки, safe-area) — параллельный канал, не трогает iOS-код на ревью.
- Проверить Maestro-сценарии (`.maestro/`) — база для регресс-тестов.

### Фаза 1 — ПОСЛЕ апрува: вирусный soft-launch (2-3 недели)
Цель — K-factor > 1 внутри офисов + D1/D7 retention.
- **Вирусный спринт (главное):** реферальная петля (`?ref=UID` + награды обеим сторонам), shareable-карточка результата (PNG/GIF), готовое Slack/Teams-сообщение «winner skips standup». (Kimi: 5.1, 5.2, 5.3.4 — собрать вместе)
- **Онбординг** (интерактивный туториал соло + обучающий бой с лёгким ботом). (Kimi 4.1 — верно, высокий приоритет)
- **Comeback bonus + weekly challenge** — дешёвый retention. (Kimi 0.4)
- **Toasts + увеличенные тач-зоны + внятные ошибки сети** — полировка. (Kimi 0.3)
- Инкрементальный рефакторинг: выносим `SoloGame.tsx` **только если** мешает вирусным правкам.

### Фаза 2 — Монетизация (после подтверждённого retention, 3-4 недели)
- **IAP косметика** (премиум-скины + паки) + **пакеты монет** — быстрый ARPU. (Kimi 3.1.A/B — согласен, это первое)
- **B2B командный тариф — тестировать РАНЬШЕ, чем у Kimi:** «Premium Team Room» ($9.99/команда — без рекламы, приватные комнаты, командный лидерборд). Это уникальный высокий LTV. (Kimi 3.1.D — но я поднимаю приоритет)
- Rewarded ads — **опционально, A/B**: только «удвоить награду», без принуждения. Смотрим, не бьёт ли по офисному UX. (Kimi 0.2 — но осторожнее)

### Фаза 3 — Масштаб (после revenue + retention, 1-3 мес)
- **Корпоративный функционал:** команды/офисы как долгоживущие сущности, командные лидерборды, офисные челленджи, турниры. (Kimi 5.3, 2.1 — ядро долгосрочной ценности)
- **Сезонный рейтинг + Battle Pass** — retention + монетизация, когда есть аудитория. (Kimi 2.2)
- **Android полноценный релиз** + оптимизация рендера party (мемоизация сегментов, отключить тени на Android). (Kimi 2.5)

### Технический трек (сквозной, параллельно)
- **Мультиплеер interpolation** (быстрый win против рывков) — когда party станет ключевым каналом. (Kimi 2.3.1)
- **Чит-защита ranked** (двойное подтверждение результата) — до серьёзного роста ranked. (Kimi 2.3.3)
- **Рефакторинг App.tsx** — инкрементально под фичи (SoloGame, Menu, useWallet). (Kimi 2.1)
- **Тесты сетевых хуков** (`useRoom.test.ts`) — до масштабирования мультиплеера. (Kimi 2.4)

## 4. Ключевое отличие моего плана от Kimi

| | Kimi | Мой синтез |
|---|---|---|
| Отправная точка | Сразу фичи + рефакторинг | Сначала апрув Apple + данные PostHog |
| Рефакторинг App.tsx | Ранний отдельный заход (Этап 1) | Инкрементально под фичи |
| Виральность | Размазана по этапам | Фокусный спринт сразу после апрува |
| B2B монетизация | Этап 2 | Тестировать раньше (уникальный актив) |
| Rewarded ads | Quick win | Опционально, осторожно (офисный UX) |
| Философия | Технически-полный план | Метрики-driven, под бизнес-реальность |
